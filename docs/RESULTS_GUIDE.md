# Results Guide

Generated files are evidence only when a documented command created them and the file was inspected. A smoke run can prove integration; it cannot by itself prove DDPG convergence.

## Result-directory contract

```text
results/
|-- checkpoints/   # model and optimizer checkpoints
|-- metrics/       # machine-readable JSON run records
|-- plots/         # learning and loss curves
|-- reports/       # generated human-readable run summaries (planned)
|-- screenshots/   # GUI screenshots (planned)
`-- trajectories/  # random and evaluated robot paths
```

Raw/generated results should remain under `results/`. If the submission must display images directly on GitHub, place reviewed copies under `assets/demo/` and label their originating command, seed, and source result path. Do not hand-edit graphs or copy a random-policy image under a trained-policy name.

## Current simulator result

Generate the implemented artifact with:

```bash
uv run robot-vacuum demo --max-steps 150 --seed 42
```

Expected file:

```text
results/trajectories/random_policy.png
```

The figure must contain non-empty path data, map obstacles, axis labels, a title, and start/final markers. Its title must continue to identify the run as random-policy evidence.

## Simulation-report schema

`results/metrics/random_policy_metrics.json` contains:

```json
{
  "schema_version": "1.00",
  "run_type": "random_policy_demo",
  "command": "uv run robot-vacuum demo --max-steps 150 --seed 42",
  "seed": 42,
  "max_steps": 150,
  "steps_executed": 150,
  "total_reward": 0.0,
  "collisions": 0,
  "coverage_percent": 0.0,
  "final_position": {"x": 0.0, "y": 0.0},
  "final_heading": 0.0,
  "map_name": "simple_house",
  "generated_artifacts": {
    "trajectory": "results/trajectories/random_policy.png",
    "metrics": "results/metrics/random_policy_metrics.json",
    "report": "results/reports/random_policy_report.md"
  }
}
```

The numeric values above illustrate the stable schema only. Each command replaces them with actual SDK results. The generated Markdown report uses the same in-memory metrics record, so it does not maintain a second set of manually entered values.

## DDPG result acceptance rules

Do not change a DDPG artifact from `Pending` until all applicable checks pass:

1. The generating actor, critic, replay, noise, target, and trainer code exists and has direct tests.
2. The exact `uv run` command, seed, resolved configuration, date, device, and duration are recorded in metrics.
3. `training_metrics.json` contains real per-episode rewards and per-update critic losses.
4. `learning_curve.png` is generated from episode cumulative reward with episode number on the x-axis.
5. `critic_loss.png` is generated from critic MSE with optimizer update index on the x-axis.
6. `evaluation_trajectory.png` is generated using a loaded checkpoint with exploration noise disabled.
7. Images are non-empty and visually inspected.
8. `docs/SUMMARY_REPORT.md` links the actual paths and distinguishes smoke, baseline, and evaluation runs.

Passing smoke training demonstrates that the pipeline works. A convergence or trained-performance claim requires a longer recorded run and supporting metrics/figures; it must not be inferred from file existence alone.

## Report synchronization checklist

Whenever artifacts are generated:

- update `docs/SUMMARY_REPORT.md` with actual paths and observations;
- update `README.md` with the reproducible command and selected reviewed images;
- update `docs/PROMPT_LOG.md` for material AI-assisted implementation or interpretation;
- update `docs/TODO.md` only after verifying the file and command;
- keep planned artifacts visibly marked `Pending`;
- run Ruff and tests again before submission.
